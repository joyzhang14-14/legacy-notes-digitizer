# Failed.md — 失败记录与教训

> 每次失败都不能白费。记录在此，防止走回头路。

---

## 失败 #1: PDF 字体不嵌入 → 全白页面
- **时间**: v1 Stage 4
- **现象**: 生成的 PDF 打开后全白，看不到任何文字
- **根因**: reportlab 使用系统字体路径（如 /System/Library/Fonts/STHeiti Medium.ttc），Mac 上路径不匹配导致字体注册失败。PDF 的 /Resources 中没有 /Font 条目。
- **教训**: 不要依赖系统字体路径。使用 pip 可安装的开源字体（Noto CJK），或在脚本中自动下载字体到项目目录。
- **解决**: v2 Phase 3 使用 macOS 系统字体 STHeiti/Songti + fallback 机制

## 失败 #2: OCR 识别为日文
- **时间**: v1 Stage 2 + Stage 4
- **现象**: 第 3 页被完全误判为日文（平假名、片假名），输出如 "予備３か月目分"、"図わち見訓服装をさい"
- **真实内容**: Gemini 正确识别为 "因为考虑到服装效果图主要突出绘画技法和表现方法"，任课教师：张巍媛
- **根因**: 
  1. API 代理可能没调用真正的 Opus 模型
  2. Prompt 中缺乏足够强的中文约束
  3. Sonnet 单轮识别对极度模糊的手写体力不从心
- **教训**: 手写中文 OCR 必须在 prompt 中反复强调"绝对没有日文"。Gemini 2.5 Pro 对手写中文明显强于 Claude Sonnet/代理 Opus。
- **解决**: v2 改用 Gemini 做 OCR，Claude 做语义校正

## 失败 #3: PPTX 打不开
- **时间**: v1 Stage 4
- **现象**: Mac 上用 Keynote 和 PowerPoint 都无法打开生成的 PPTX
- **根因**: 字体引用问题。python-pptx 引用了系统中不存在的字体名。
- **教训**: PPTX 中使用的字体名必须和系统中实际安装的字体名完全一致。
- **解决**: v2 自动安装 Noto CJK 字体到 ~/Library/Fonts/

## 失败 #4: 统一预处理导致插图细节丢失
- **时间**: v1 架构设计缺陷
- **现象**: 对整页做 CLAHE + 二值化后，插图的灰度渐变、淡铅笔线被吞掉
- **教训**: 插图和文字需要完全不同的对比度处理。应该先识别结构，再分通道处理。
- **解决**: v2 双通道架构（插图保留原始对比度，文字单独增强）

## 失败 #5: API 代理不稳定
- **时间**: 贯穿整个项目
- **现象**: 频繁出现 503（渠道失败）、524（超时）错误。Claude Code 本身也受影响。
- **错误信息**: "尝试[CC官方渠道]渠道后均失败" / "524: A timeout occurred"
- **教训**: 长时间任务不要依赖不稳定的代理。脚本必须有完善的重试机制和断点续跑。
- **解决**: Gemini 用 Google 官方 API，Claude 用 .env 配置可切换

## 失败 #6: Opus 太慢 + 太贵
- **时间**: v2 Phase 1 + Phase 2b
- **现象**: Phase 1 双轮 Opus 扫描耗时 80s/页，Phase 2b 三轮 OCR 耗时 415s/页（接近 7 分钟/页）。100 页预计需要 12 小时。
- **教训**: Opus 适合做最终校验，不适合做主力工作。Sonnet 做结构扫描足够了。
- **解决**: 新架构中结构扫描用 Sonnet，OCR 用 Gemini，只在交叉校验时用 Claude

## 失败 #7: 图片超过 5MB 限制
- **时间**: v2 Phase 1
- **现象**: page_002 的 300dpi PNG 为 5.9MB，超过 Claude API 的 5MB 限制
- **错误**: "image exceeds 5 MB maximum: 5905112 bytes > 5242880 bytes"
- **教训**: 300dpi 的大页面 PNG 很容易超 5MB。发送前必须检查文件大小，超出时压缩或缩小。
- **解决**: 发送前自动检查，超过 4.5MB 时缩小到长边 3000px 或转为 JPEG quality=85

## 失败 #8: 孤立文字区域 OCR → 断句不正常
- **时间**: v2 Phase 2b
- **现象**: 将文字区域单独裁剪后发给 Gemini OCR，识别结果是一段一段无联系的话。缺少上下文导致语义断裂。
- **根因**: 结构扫描（Phase 1）把文字切成了独立小块，每块单独 OCR 时模型看不到上下文。
- **教训**: OCR 应该发送整页图片（增强后），让模型在完整上下文中识别。位置信息由结构扫描提供，OCR 只负责内容识别。
- **解决**: 新架构：Sonnet 做位置编码 → 整页增强 → Gemini 看整页做 OCR → 位置重映射

## 失败 #9: Claude 交叉校验过度修正
- **时间**: v2 Phase 2b Round 3
- **现象**: Claude 校验时把 Gemini 的正确识别"教孙子"改成了"裁故子"（其实原文更可能是"教案"的某个变体）
- **根因**: 专业术语词典太大，Claude 倾向于把任何不确定的字都往术语上靠
- **教训**: 交叉校验应该保守——只修正有高置信度的错误，不确定的保留 Gemini 的结果
- **解决**: Round 3 prompt 中强调"如果不确定，保留 Gemini 的原始结果"

---

## 架构层面的教训总结

1. **先识别再处理** — 不要在看清楚之前就动手改图
2. **整页上下文** — OCR 不能切碎了喂，要给模型完整的页面
3. **快模型做主力，强模型做校验** — Sonnet 做结构，Gemini 做 OCR，Claude 做最终校正
4. **特殊符号用 trace 不用 OCR** — 圈重点、马克笔、箭头等直接 trace 为 SVG
5. **颜色判断基于粗细不是颜色本身** — 同色线条，粗 3x 的是马克笔
6. **文件大小要检查** — API 有限制，300dpi 大图容易超
7. **代理不可靠** — 关键 API 调用必须有重试和降级机制
